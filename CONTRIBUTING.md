# Contributing to VSRMS

First off, thank you for considering contributing to VSRMS! It's people like you that make VSRMS such a great tool.

## How to Contribute

### 1. Report Bugs
If you find a bug, please create an issue with:
* A clear title and description
* Steps to reproduce the issue
* Expected vs actual behavior
* Your environment (OS, Node version, Expo version)

### 2. Suggest Features
We welcome feature requests! When suggesting a feature:
* Explain the use case and why it would be beneficial
* Provide examples or mockups if possible

### 3. Code Contributions
If you want to contribute code:
1. Fork the repository
2. Create a feature branch (`git checkout -b feature/amazing-feature`)
3. Write clean, documented code following the project's style guidelines
4. Ensure your code passes all linting and type checks
5. Commit your changes (`git commit -m 'feat: add amazing feature'`)
6. Push to the branch (`git push origin feature/amazing-feature`)
7. Open a Pull Request

## Development Setup
Please refer to the [Quick Setup Guide](docs/setup.md) for detailed instructions on setting up your local development environment.

## Code Style
* Use TypeScript for all new mobile and shared code.
* Use Zod for validation schemas where possible.
* Maintain the Feature-Sliced Context architecture in the mobile app.
* Ensure no secrets are hardcoded in the repository.

Thank you for your contributions!
